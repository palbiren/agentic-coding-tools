Parallel Agentic Development

Multi-Agent Coordination for AI-Assisted Software Engineering

Agent Coordinator Architecture & Implementation

Overview of the two-level parallel development system and the coordinator that makes it possible.

The Problem

What happens when multiple AI agents work on the same codebase?

  • Concurrent edit conflicts — two agents modify the same file
  • Context loss — sessions end, context disappears
  • Task duplication — agents do redundant work
  • Unsafe operations — destructive commands without oversight
  • No visibility — who is doing what, where?
These are real problems we hit as soon as you go beyond a single agent session. The coordinator exists to solve all five.

Three-Level Coordination

Level 3 (human): portfolio prioritization, epic sequencing, approval gates. Humans retain all strategic decisions. Level 2 (coordinator-assisted): feature registry detects conflicts, merge queue orders merges. Level 1 (agent-automated): DAG scheduling, lock protocol, scope enforcement — fully automated. Automation increases as scope narrows.

Level 2: Cross-Feature Parallelism

Feature Registry

  • Features register resource claims before implementation
  • Conflict analysis detects overlaps
  • Feasibility: FULL PARTIAL SEQUENTIAL

Merge Queue

  • Priority-ordered merging
  • Pre-merge conflict re-validation
  • Automatic rebase coordination
  • States: QUEUED → PRE_MERGE → READY → MERGED
FULL = no overlapping claims, safe to parallelize completely. PARTIAL = some overlap, needs coordination (e.g., shared DB schema). SEQUENTIAL = >50% overlap, must serialize.

Level 1: Intra-Feature Parallelism

Work Packages

  • Feature decomposes by architectural boundary
  • Each package: own agent, worktree, scope
  • DAG dependencies between packages
  • Contracts → Backend → Frontend

Key Invariants

  • One agent, one worktree, one branch
  • Scope enforcement via write_allow / deny globs
  • Deterministic scope check: git diff vs allowed files
  • Deadlock-safe lock ordering
"Inverse Conway Maneuver" — agents organized by architectural boundary, not lifecycle phase. Scope enforcement is deterministic: git diff output is checked against globs after code generation.

Level 3: Program Coordination

Human Approval Gates

  • Feature selection — what to build next
  • Plan approval — review proposal before impl
  • Code review — approve PR before merge
  • Validation sign-off — accept deployment
  • Security escalation — human-only decisions

Cross-Application Patterns

  • Epic tracking via Beads issue tracker
  • /prioritize-proposals for portfolio ranking
  • Coordinator memory persists across sessions
  • OpenSpec specs as shared source of truth
  • Strategic decisions stay with humans
Level 3 is intentionally human-driven. The system optimizes for tactical parallelism, not strategic portfolio management. Humans make all "what to build" decisions. Automation increases as scope narrows: Level 3 (human) → Level 2 (assisted) → Level 1 (automated).

Parallel Workflow

Falls back to linear single-agent mode when coordinator unavailable

7 parallel skills mirror the linear workflow but add: resource claim analysis, contracts, work-packages.yaml, DAG scheduling, vendor-diverse reviews, merge queue. Falls back to linear when coordinator is unavailable — graceful degradation.

Phase A: Feature-Level Preflight

Orchestrator registers, then submits all work packages with dependency edges. The DAG determines execution order — packages with no deps are immediately claimable.

Phase B: Parallel Package Execution

Contracts complete first (no deps). Then backend + frontend run truly in parallel. Each agent: claim work, acquire locks, read dependency results, implement, scope check, verify, complete. Phase C follows: merge worktrees, run integration suite, submit to merge queue.

Lock Acquisition: Happy Path

Key insight: all agents sort locks in the same lexicographic order. This guarantees no circular wait — the textbook deadlock prevention technique.

Lock Conflict: Rollback + Retry

On conflict: release ALL held locks in reverse order, backoff with jitter, retry (max 3). After 3 failures: FAIL with LOCK_UNAVAILABLE, escalate to orchestrator.

Namespaces give semantic meaning to locks beyond just file paths. The coordinator doesn't interpret prefixes — it's all opaque strings with lexicographic ordering. The pause lock is the "stop-the-line" mechanism for the orchestrator.

Merge Queue: Registration

Features register resource claims before implementation. Conflict analysis detects overlapping claims and assesses feasibility. PARTIAL means features can run in parallel but need coordination during merge.

Merge Queue: Priority Merging

Features merge in priority order with pre-merge re-validation. Lower-priority features rebase automatically after each merge.

Guardrails & Trust Model

Every task claim passes through guardrails before dispatch. Pattern matching detects: git force-push, rm -rf, DROP TABLE, env file access. Trust levels gate operations: standard agents (level 2) can't run admin ops (level 3+). All decisions are audit-logged immutably.

11-Step Worker Protocol

This is the contract every parallel worker follows. Key safety points: - B2/B9: Pause checks let the orchestrator halt all workers instantly. - B3: Lexicographic lock ordering prevents deadlocks. - B7: Deterministic scope check ensures agents stay within their allowed files. - B11: Cleanup is guaranteed — ports and locks are always released.

Contract-First Development

Contracts are the coordination boundary between parallel agents. The contracts package runs first, generates types and mocks. Backend and frontend agents can develop against mocks independently. Integration verification catches mismatches via property-based and consumer-driven tests.

Transport & Graceful Degradation

CLI agents use MCP (stdio) — zero network config needed. Cloud agents use HTTP API with API-key auth. Skills detect coordinator availability and degrade: Full capabilities → parallel skills. No coordinator → automatic fallback to linear single-agent workflow.

Escalation Protocol

Blocking

  • CONTRACT_REVISION_REQUIRED
    Bump revision, regenerate, resubmit
  • PLAN_REVISION_REQUIRED
    Modify DAG, increment plan_revision
  • SCOPE_VIOLATION
    Fail package, require plan revision
  • SECURITY_ESCALATION
    Pause DAG, require human approval

Non-Blocking

  • RESOURCE_CONFLICT
    Retry with backoff
  • VERIFICATION_INFEASIBLE
    Reassign or create follow-up
  • ENV_RESOURCE_CONFLICT
    Fresh port allocation
  • FLAKY_TEST_QUARANTINE
    Quarantine, re-run, don't block

Stop-the-Line: Orchestrator acquires feature:<id>:pause lock. Workers check at B2 and B9.

8 escalation types with deterministic decisions. The pause-lock is the kill switch — instantly halts all workers on a feature. Security escalations always require human approval.
